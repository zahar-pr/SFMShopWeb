# Архитектура гибридной БД для SFMShop

## Распределение данных

### PostgreSQL
Основное хранилище бизнес-данных:
- `users` — пользователи
- `products` — товары
- `orders` — заказы
- `order_items` — состав заказа
- `reviews` — отзывы

**Почему:**  
Реляционные связи, транзакции ACID, целостность данных, внешние ключи.

---

### Redis
Быстрые временные данные:
- `cache:products`
- `cache:product:{id}`
- `session:{user_id}`
- `queue:notifications`

**Почему:**  
Очень быстрый доступ, TTL, удобно для кэша, сессий и очередей.

---

### MongoDB
Неструктурированные и событийные данные:
- `logs`
- `user_events`
- `analytics`

**Почему:**  
Гибкая схема документов, удобно хранить логи, действия пользователей и аналитику.

---

## Поток данных

1. При запросе товара:
   - сначала Redis
   - если нет, то → PostgreSQL
   - затем запись в Redis

2. При создании заказа:
   - запись в PostgreSQL
   - очистка кэша Redis
   - запись события в MongoDB

3. При входе пользователя:
   - проверка пользователя в PostgreSQL
   - создание сессии в Redis
   - лог события в MongoDB

---

## Компромиссы

**Плюсы:**
- высокая производительность
- правильное использование сильных сторон каждой БД
- хорошая масштабируемость

**Минусы:**
- сложнее поддержка
- нужно следить за консистентностью между хранилищами
- больше инфраструктурных компонентов

orm - способ взаимодействия с бд с помощью объектов, методов и классов языка программирования. N+1 проблема - lazy loading когда используем ленивую подгрузку данных - 100 строк ~ 100 запросов к базе увеличивает latency. 
решения в алхимии eager loading
selectinload -  делает 2 запроса в базу, первый за основными объектами, второй за связями через WHERE | one to many
joinedload - подгружает все связи через left join с помощью одного запроса one to one или many to one
проблема
users = session.query(User).all()

for user in users:
    print(user.orders)
решение
users = session.query(User).options(
    joinedload(User.orders)
).all()

for user in users:
    print(user.orders)
